Access Flow

From approved
to inside — flawlessly.

✅
STEP 01

Luma Approval

Event organizer pre-approves every attendee. No surprises at the door — the guest list is locked before the event starts.

→
🔁
STEP 02

Auto-Sync

Ipê Fences pulls the confirmed list from Luma automatically. Residents no longer need to be re-added every month — the system handles it.

→
📲
STEP 03

One-Time QR Token

Each attendee receives a cryptographic QR code valid for that event only. Single-use, time-bound, non-transferable — invalidated immediately on scan.

→
🚪
STEP 04

Door Unlocks

Validated identity → door opens in under 2 seconds. Entry is logged, the smart home is notified, and the dashboard updates in real time.

// Token lifecycle
token.generate({ event_id, member_id, expires_at }) // issued on approval
token.validate(qr_scan) → { valid: true, uses_remaining: 0 } // consumed on entry
token.validate(qr_scan) → { valid: false, reason: "already_used" } // rejected on retry
Security

Designed to be
unforgeable.

🔑

Single-Use Tokens

Every QR is cryptographically bound to one person and one event. The moment it's scanned, it's dead. Replay attacks are structurally impossible.

📍

Timestamped Audit Log

Every entry is logged with identity, timestamp and event context. Full traceability without exposing unnecessary personal data.

✅

Pre-Approved Lists

Nobody walks in unvetted. Luma's approval flow is the gate — entry is only possible if the organizer explicitly confirmed the person beforehand.

🔗

Verified Identity Chain

Each token is linked to a member profile — creating an unambiguous chain from identity → approval → token → physical entry.

⚡

Real-Time Alerts

Anomalies — invalid tokens, duplicate scans, unexpected spikes — trigger instant notifications to the ops team via Telegram webhooks.

🛡️

Data Minimization

The system logs only what's operationally necessary. No behavioral profiling, no third-party telemetry — data stays within the community's own infrastructure.

Privacy

Your data.
Your rules.

For a community built on trust, privacy isn't a feature — it's the foundation.

PRINCIPLE 01

Community-Owned Infrastructure

All data lives in infrastructure the community controls. No vendor lock-in, no third-party data harvesting. Members are users, not the product.

PRINCIPLE 02

Minimum Necessary Data

We collect what the system operationally needs — and nothing more. No behavioral analytics, no ad targeting, no surveillance layer.

PRINCIPLE 03

Open for Audit

The access log is fully auditable by authorized community members. Transparency where it matters — opacity where it's warranted.

PRINCIPLE 04 · ROADMAP

On-Chain Attestation Layer

Access events as signed, verifiable claims — enabling portable membership credentials, composable identity, and trustless verification without exposing raw data.
